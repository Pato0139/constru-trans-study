1. ¿Qué es Constru-Trans?

Constru-Trans es un sistema integral de gestión para transporte y materiales de construcción. Combina gestión de inventario, pedidos, facturación, pagos, transportistas y reportes en una sola plataforma.

2. ¿Qué problema de negocio resuelve?

  • ✅ Gestionar materiales de construcción y su stock
  • ✅ Controlar pedidos de clientes
  • ✅ Facturar y registrar pagos (parciales o totales)
  • ✅ Gestionar transportistas y vehículos
  • ✅ Generar reportes en PDF, Excel y XML
  • ✅ Trabajar offline sin internet

3. Arquitectura híbrida Offline-First ⭐

Este es el punto clave único del proyecto:

📦 BD Local: SQLite

Archivo db.sqlite3 en cada PC. Funciona sin internet.

  • Trabajas normalmente sin conexión
  • Todos los datos están en tu PC
  • No depende de la nube para funcionar

☁️ BD Remota: Neon PostgreSQL

Base de datos central en la nube. Sincroniza usuarios, sesiones y historial entre compañeros.

  • Compartir datos entre el equipo
  • Centralizar credenciales de login
  • Historial común para todos

Router de BD: El archivo core/routers.py decide automáticamente qué base de datos usar (local o remota) según la operación. ¡Genial!

5. ¿Qué es un entorno virtual?

Imagina que tienes dos proyectos:

  • Proyecto A necesita Django 4.2
  • Proyecto B necesita Django 5.0

Si instalas Django en tu ordenador "a secas", ambos proyectos usarán la misma versión y habrá conflictos.

Un entorno virtual es como una "caja separada" para cada proyecto, con sus propias versiones de Python y paquetes. ¡Nunca hay conflictos!

📚 Lección 3: Django Básico — Cómo crear un proyecto y la carpeta core

🎯 Qué vas a aprender

  • Cómo crear un proyecto Django desde cero
  • Qué es la carpeta core (o el nombre del proyecto)
  • Cómo se estructura un proyecto Django
  • Relación con tu repositorio real

1. Instalar Django

Primero, con el entorno virtual activado, instala Django:

pip install django

2. Crear un proyecto Django

Para crear un proyecto Django, usa el comando:

django-admin startproject nombre_del_proyecto

En tu repositorio real, el proyecto se llama Constru-trans, y la carpeta principal se llama core. Esto se hace así:

django-admin startproject core .

⚠️ El punto . al final es importante! Significa "crear el proyecto en la carpeta actual", sin crear una subcarpeta extra.

3. ¿Qué archivos se crean?

Después de ejecutar el comando, tendrás esta estructura:

tu_carpeta/ ├── manage.py # Archivo de gestión (para runserver, migrate, etc.) └── core/ # Carpeta del proyecto (tú puedes cambiarle el nombre) ├── __init__.py # Indica que esta carpeta es un paquete Python ├── settings.py # Configuración del proyecto ├── urls.py # Rutas principales └── wsgi.py # Para部署 (deployment) — no lo usas mucho en desarrollo

4. Relación con tu repositorio real

En tu repo Constru-trans_01, la estructura es:

Constru-trans_01/ ├── manage.py ├── core/ # Carpeta del proyecto │ ├── settings/ # ⭐ Configuración modular (base/dev/prod) │ ├── urls.py # Rutas principales │ └── routers.py # Router de BD (local vs remoto) └── apps/ # Todas las apps del proyecto ├── usuarios/ ├── clientes/ ├── ordenes/ └── ...

Diferencia clave: Tu repo tiene una carpeta core/settings/ modular (con archivos separados para base, desarrollo y producción), en lugar de un solo settings.py. Esto es una buena práctica para proyectos grandes.

5. Archivos clave en core/

  • core/urls.py: Rutas principales del proyecto (incluye todas las apps)
  • core/settings/: Configuración (base de datos, apps instaladas, etc.)
  • core/routers.py: Router de BD (decide si usa SQLite local o PostgreSQL remoto)

🧠 Qué debes memorizar

  • django-admin startproject core .: Crea un proyecto Django en la carpeta actual, con la carpeta del proyecto llamada core
  • manage.py: Archivo para ejecutar comandos (runserver, migrate, createsuperuser)
  • La carpeta core/ contiene la configuración del proyecto

❌ Error común

Olvidar el punto . al final del comando: django-admin startproject core (sin punto) crea una carpeta extra. ¡Siempre usa el punto!

Paso 1: ¡Empecemos por lo más básico! ¿Qué es una URL?

Imagina esto:

Una URL es la dirección que escribes en la barra del navegador para ir a un lugar en internet. Por ejemplo:

  • https://www.google.com → la URL de Google
  • http://127.0.0.1:8000/usuarios/login/ → la URL para iniciar sesión en tu proyecto

Paso 2: ¿Qué son las "rutas" en Django?

Analogía super simple:

Imagina que tu proyecto Django es un edificio con varias habitaciones:

  • El recepción del edificio es el archivo core/urls.py
  • Cada habitación es una app (usuarios, clientes, facturacion, etc.)
  • Las rutas son las flechas que te dicen cómo llegar a cada habitación

En palabras sencillas:

Las rutas le dicen a Django: "Si alguien visita esta dirección, haz esto".

Paso 3: El archivo más importante: core/urls.py

TODAS las peticiones a tu proyecto pasan primero por aquí. Es la recepción del edificio.

Dentro de core/urls.py hay una lista llamada urlpatterns

Esta lista es como un libro de direcciones: cada línea dice "si viene alguien por aquí, llévalo a esta app".

Paso 4: 2 palabras clave que debes saber: path() e include()

1. path() — "Crea una nueva dirección"

Imagina que path() es como pegar un letrero en la pared que dice: "Por aquí se va a X lugar".

Ejemplo:

path('usuarios/', ...)

Esto significa: "Si alguien escribe /usuarios/ en el navegador, haz lo que viene a continuación".

2. include() — "Ve a buscar más direcciones en otra carpeta"

include() es como decir: "No sé todas las direcciones de esta habitación, ve a preguntar a la app correspondiente".

Ejemplo:

path('usuarios/', include('apps.usuarios.urls'))

Esto significa: "Si alguien viene por /usuarios/, ve a la app 'usuarios' y mira su archivo urls.py para saber qué hacer".

Paso 5: Veamos un ejemplo REAL y SIMPLE

# Esto es lo que hay en core/urls.py from django.urls import path, include urlpatterns = [ # 1. Si alguien va a /admin/, usa el admin de Django path('admin/', admin.site.urls), # 2. Si alguien va a /usuarios/, ve a la app de usuarios path('usuarios/', include('apps.usuarios.urls')), # 3. Si alguien va a /clientes/, ve a la app de clientes path('clientes/', include('apps.clientes.urls')), # 4. Si alguien va a la página principal (solo /), ve a la app de inicio path('', include('apps.inicio.urls')), ]

Paso 6: ¡Vamos a seguir el flujo de una petición! 🚶‍♂️

Imagina que alguien escribe en el navegador: http://127.0.0.1:8000/usuarios/login/

Esto es lo que pasa PASO A PASO:

  1. El navegador envía la petición a Django.
  2. Django va a la recepción: core/urls.py.
  3. Django ve que la URL empieza por 'usuarios/'.
  4. Django busca en la lista urlpatterns y encuentra: path('usuarios/', include('apps.usuarios.urls')).
  5. Django dice: "Ah, ok, las direcciones de usuarios están en apps/usuarios/urls.py — voy allá".
  6. Django abre apps/usuarios/urls.py y busca la parte restante de la URL: 'login/'.
  7. Django encuentra la ruta para 'login/' y ejecuta la vista correspondiente (el código que muestra la página de login).
  8. ¡Listo! El usuario ve la página de inicio de sesión.

Código real de tu repositorio (así es en tu proyecto!)

urlpatterns = [ path('admin/', admin.site.urls), path('licensing/', include('apps.licensing.urls')), path('usuarios/', include('apps.usuarios.urls')), path('clientes/', include('apps.clientes.urls')), path('inventario/', include('apps.inventario.urls')), path('compras/', include('apps.compras.urls')), path('ordenes/', include('apps.ordenes.urls')), path('facturacion/', include('apps.facturacion.urls')), path('pagos/', include('apps.pagos.urls')), path('', include('apps.inicio.urls')), ]

Épica 1 · Modelos base

La autenticación real parte del modelo nativo User de Django, pero el proyecto lo amplía con Usuario. Ese modelo añade nombres, apellidos, documento, teléfono y el campo más importante para permisos: rol.

Luego aparecen perfiles especializados. Conductor extiende a un usuario con licencia y EPS. Cliente cuelga del usuario con dirección, tipo y datos comerciales. Así el proyecto separa credenciales del dominio del negocio.

Código real

class Usuario(models.Model): user = models.OneToOneField(User, on_delete=models.CASCADE, related_name="usuario") nombres = models.CharField(max_length=200) apellidos = models.CharField(max_length=200) telefono = models.CharField(max_length=20, blank=True) documento = models.CharField(max_length=20) rol = models.CharField(max_length=50, choices=ROLES) class Conductor(models.Model): usuario = models.OneToOneField(Usuario, on_delete=models.CASCADE, primary_key=True, related_name='perfil_conductor') numero_licencia = models.CharField(max_length=50, unique=True) class Cliente(models.Model): usuario = models.OneToOneField('usuarios.Usuario', on_delete=models.CASCADE, primary_key=True, related_name='perfil_cliente')

Esta es la parte más delicada. La vista valida monto, método de pago, permisos, estado de la factura y saldo pendiente. Luego bloquea la factura con select_for_update() dentro de una transacción atómica para evitar concurrencia rota.

Épica 6 · Pago, estado e historial

Una factura puede tener varios pagos gracias al related_name='pagos'. El modelo `Factura` calcula total_pagado y saldo_pendiente. Además, el modelo `Pago` tiene una señal `post_save` que marca la factura como pagada si ya no queda saldo.

Código real

class Factura(models.Model): @property def total_pagado(self): return sum(p.monto for p in self.pagos.all()) @property def saldo_pendiente(self): return self.total - self.total_pagado class Pago(models.Model): factura = models.ForeignKey(Factura, on_delete=models.CASCADE, related_name='pagos') @receiver(post_save, sender=Pago) def actualizar_estado_factura(sender, instance, created, **kwargs): if created: factura = instance.factura if factura.saldo_pendiente <= 0 and factura.estado != 'pagada': factura.estado = 'pagada' factura.save()

Paso 1: ¿Qué es el MER?

Objetivo

Entender qué es un Modelo Entidad-Relación y por qué es útil.

¿Qué es?

El MER es un diagrama que muestra las entidades (tablas) del sistema, sus campos y las relaciones entre ellas.

Relaciones importantes:

  • 1:1 (Uno a uno): Un registro de A se relaciona con UN solo registro de B (ej: User ↔ Usuario)
  • 1:N (Uno a muchos): Un registro de A se relaciona con VARIOS registros de B (ej: Pedido ↔ DetallePedido)
  • N:M (Muchos a muchos): Varios registros de A se relacionan con varios de B (ej: Conductor ↔ Vehiculo, tabla puente ConductorVehiculo)
